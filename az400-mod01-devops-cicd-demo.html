🔷 MODULE 1 — DevOps & CI/CD
DevOps Core Concept
💡 DevOps = Culture + Process + Tools — Breaking down silos between developers and operations teams to deliver software faster and more reliably.

DevOps represents a fundamental shift from traditional waterfall development where Dev and Ops teams worked in complete isolation.

Azure DevOps vs GitHub: Feature Comparison
Feature Area Azure DevOps GitHub
Planning & Tracking Azure Boards (Epic/Feature/Story/Task) GitHub Issues, Projects, Milestones
Source Control Azure Repos (Git) GitHub Repositories
CI/CD Azure Pipelines (YAML/UI) GitHub Actions (Workflow)
Artifacts Azure Artifacts (NuGet, npm, Maven, Python) GitHub Packages
Security Limited security features Advanced Security (SAST, Dependabot)
Testing Azure Test Plans (full ALM) Manual testing, integration with CI
Best For Enterprise, Microsoft stack, complex governance Open source, modern development, smaller teams
Cost Model Per user/month (Azure subscribers) Free public, paid private repos
Use Azure DevOps When:
• Large enterprise teams (100+ developers)
• Heavy Microsoft stack (C#, .NET, SQL Server)
• Complex ALM requirements (Test Plans, Release Gating)
• Strict governance and compliance needs
• Multiple project coordination
• On-premises or hybrid deployments
Use GitHub When:
• Open source projects
• Distributed teams / community
• Fast-moving startups
• Advanced security scanning needed
• Cloud-native / modern stack
• Lower operational overhead desired

Testing Pyramid: Types of Tests
Unit Tests
Test individual functions/methods in isolation. Run every commit. 70% of tests.
Integration Tests
Test interactions between modules/services. Run on PR. 20% of tests.
End-to-End Tests
Test full user workflows. Run in staging. 10% of tests.

Common Exam Traps
Testing is only important at the end of development. CI/CD is just about deployment speed.
Testing throughout the development cycle catches bugs early, reducing overall cost. CI/CD enables continuous improvement and quality.
Azure DevOps and GitHub are competing tools — choose one or the other.
Many organizations use both together. GitHub for coding, Azure Boards for planning and Azure Artifacts for packages.
A 100% passed test rate means your code is production-ready.
High test coverage is good, but you need the right kinds of tests (unit, integration, E2E). Also, tests can have bugs.
Release and Deployment are the same thing.
Release = code is ready to deploy. Deploy = code goes live. Release happens in staging; deployment happens to production.
Key Formulas & Metrics to Remember
Deployment Frequency
How often code is deployed to production. High = good (enables rapid feedback). Target: daily or more.
Lead Time
Time from code commit to production. Lower = better. Target: hours, not days.
Change Failure Rate
Percentage of deployments causing incidents. Lower = better. Target: <15%.
Mean Time to Recovery (MTTR)
Time to fix a production issue. Lower = better. Target: minutes to hours.
Pro Tip: The exam loves scenario questions. "A team needs to reduce deployment risk and wants to test code early. Which practice aligns?" Answer: Shift-left testing + automated testing in CI pipeline.
