What is InnerSource?

Four Principles of InnerSource

Openness
Source code is readable by all internal developers. Anyone can propose changes via PRs. Discussions are transparent.
Transparency
Project roadmap, decisions, and issues are visible. Why was something rejected? Anyone can see the rationale.
Voluntary Code Contribution
Contributing is optional, not mandated. Teams contribute when it benefits them. Quality over quantity.
Prioritized Mentorship

Senior developers (Trusted Committers) mentor new contributors. Knowledge transfer is a priority, not an afterthought.

Real-World Impact (Anonymized Examples)

  • Large Bank: Reduced internal library duplication from 47 to 8 core libraries through InnerSource. Saved millions in maintenance.
  • Tech Firm: Built internal "npm" registry. Teams now share 300+ reusable packages. 40% faster feature development.
  • Automotive Company: Cross-team InnerSource contributions prevented 3 major security incidents (shared code review caught 0-day patterns).

Syncing Fork with Upstream

Your fork gets out of sync with the upstream repo. Here's how to update it:

# Add upstream as a remote (one-time)
git remote add upstream https://github.com/org/original-repo.git

# Fetch latest from upstream
git fetch upstream

# Rebase your changes on top of upstream/main
git rebase upstream/main

# Force push to your fork (safe because it's your repo)
git push --force origin main
⚠️ Exam Trap: The question "How do you keep a fork synchronized?" tests whether you know git fetch upstream and git rebase. Some candidates try git pull which creates merge commits (messy).

Common Fork Workflow Scenarios

Scenario 1: You Forgot to Create a Branch
# You committed directly to main. Oops.
git log --oneline
a1b2c3d Fix login bug
d4e5f6g Initial commit

# Create new branch from your commit
git branch feature/login-fix
git reset --hard upstream/main
git checkout feature/login-fix
git push origin feature/login-fix
Scenario 2: Your PR Got Conflict Notifications
# Rebase to resolve conflicts
git fetch upstream
git rebase upstream/main
# Fix conflicts in your editor
git add .
git rebase --continue
git push --force origin your-branch
✅ Best Practice: Always keep your fork updated with git fetch upstream frequently. This prevents conflict explosions.

Trusted Committers vs Contributors

Role Definitions

Contributor

Who: Any internal developer

Can Do:

  • Fork the repo
  • Create feature branches
  • Submit PRs
  • Discuss in issues
  • Review others' PRs (advice)

Cannot Do:

  • Merge PRs
  • Push directly to upstream
  • Delete branches
  • Release versions
Trusted Committer

Who: Senior devs (usually from original team)

Can Do:

  • Review and merge PRs
  • Push directly to main
  • Manage releases & versioning
  • Mentor contributors
  • Set project direction
  • Manage access & permissions

Responsibilities:

  • Code quality
  • Maintain project health
  • Mentoring new contributors

2 Fork & Setup — Clone locally, add upstream remote

TC Benefits: Reducing Bus Factor

Bus Factor Problem

Without InnerSource (Traditional): Only 1-2 developers know critical library. If they leave, team is stuck.

With InnerSource (Trusted Committers): 5-10 TCs from different teams know the codebase. Knowledge is distributed. Single person leaving doesn't break continuity.

✅ Bus factor increases from 1-2 to 5-10

✅ Exam Focus: Understand that Trusted Committers are not just code reviewers — they're mentors, gatekeepers of quality, and knowledge sharers. Their role prevents silos.

Governing Contributions

PR Review SLAs (Service Level Agreements)

Example SLA Policy
  • Acknowledgment: TC must acknowledge PR within 1 business day (even if just "thanks, will review")
  • First Review: Substantive feedback within 3 business days
  • Merge Decision: Yes/No/Needs Changes within 5 business days
  • Abandoned PRs: If no activity for 30 days, auto-close with comment
  • Security Issues: 1 business day acknowledgment, fix within 7 days

Why SLAs Matter: Contributors need confidence they'll get feedback. Vague timelines kill participation.

Contribution Agreement

Some organizations require contributors to sign an agreement:

// CONTRIBUTOR AGREEMENT (excerpt)
I certify that:

1. I have the right to contribute this code
2. The code is my own work (or properly attributed)
3. The code does not contain trade secrets
4. The code follows company security policies
5. I grant the company license to use my contributions
6. I waive any rights to the contributed code
ℹ️ Note: Most internal InnerSource agreements are simpler than external open source CLAs (Contributor License Agreements). You're just confirming you have the right to contribute.

Quality Gates for Merging

Code Quality
  • Tests must pass (100% required)
  • Code coverage maintained (or improved)
  • Linting/style checks pass
  • No hardcoded secrets
  • No dependency vulnerabilities
Documentation
  • Changes documented
  • API docs updated
  • README/CHANGELOG updated
  • Migration guide (if breaking)
  • Examples provided
Review Requirements
  • At least 1 Trusted Committer approval
  • No unresolved comments
  • Conversation tone is respectful
  • PR description explains "why"
  • Closes related issues

Handling Unwanted Contributions Gracefully

Scenario: TC Must Reject a PR
// WRONG: Dismissive rejection
"No. This is bad."

// RIGHT: Constructive feedback
"Thanks for the contribution! This approach conflicts with our
architecture because [reason]. Here's the design pattern we use:
[example]. Would you be interested in tackling [different issue]
instead? We'd love your help there."

Key Principles:

  • Thank the contributor for their time
  • Explain why the approach doesn't work
  • Suggest an alternative or redirect them
  • Leave the door open for future contributions

Conflict Resolution & Escalation

Escalation Path

Level 1: Contributor and TC discuss in PR comments

Level 2: Involved TC + another TC make decision together

Level 3: Escalate to InnerSource Program Manager

Level 4: Team management/leadership decides

Principle: TC decision is final at Levels 1-3. Only escalate when values/culture are at stake.

Exam Tips for InnerSource (AZ-400)

💡 Tip 1: InnerSource is NOT about making code public. It's about sharing code internally with open source practices. Licensing is "Internal Use Only," not MIT/Apache.
💡 Tip 2: Fork Workflow vs Feature Branch Workflow — EXAM WILL DISTINGUISH THESE. Know when to use each.
Question Pattern Expected Answer
"How do external teams contribute to a shared library?" Fork workflow with PR to upstream
"How do you keep your fork in sync?" git fetch upstream && git rebase
"What's a Trusted Committer?" Code reviewer + mentor + gatekeeper
"How do you measure InnerSource success?" Cross-team PRs, contributor count, reuse rate
"What should a repo have to be InnerSource-ready?" README, CONTRIBUTING.md, CI/CD, good labels
⚠️ Trap 1: "InnerSource is like open source but internal." This is partially true but miss the point. InnerSource is about REUSE and COLLABORATION. It solves the "Team A doesn't know Team B built the same thing" problem.
⚠️ Trap 2: Confusing fork workflow with feature branches. Exam scenario: "Guest developer from Finance team needs to contribute to Platform team's library." Answer: Fork workflow, NOT feature branch (they don't have write access).
⚠️ Trap 3: Thinking TCs are just code reviewers. They're mentors, gatekeepers of quality, and knowledge sharers. TC absence = single point of failure (high bus factor).
ℹ️ Scenario: "Your company has 50 teams. Each team built their own logging library. Now you're implementing InnerSource. What's the first step?" Answer: Create a centralized logging library, migrate teams to use it, establish TCs, publish to InnerSource portal.
✅ Key Concepts to Master: InnerSource benefits (reuse, reduce duplication), Fork workflow (not feature branches), Trusted Committer roles, Portal/discovery, Contribution governance.
