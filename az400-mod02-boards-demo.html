Process Templates Comparison
Template Best For Work Item Types
Agile Fast-moving, cross-functional teams Epic, Feature, User Story, Task, Bug
Scrum Fixed 2-week sprints, velocity tracking Epic, Feature, Product Backlog Item, Task, Bug
CMMI Regulated industries, strict governance Epic, Feature, Requirement, Task, Bug, Review, Issue
ℹ️ Nesting Rules: Epic contains Features, Features contain User Stories, User Stories contain Tasks/Bugs. You can't nest a Task under an Epic directly.

GitHub Project Automation
GitHub Project Rules
Rule 1: When PR merges → close related issue
IF pull_request.merged THEN issue.status = "Done"
Rule 2: New issue → add to backlog
IF issue.created AND NOT assigned THEN project.status = "Todo"
Rule 3: When marked as draft → move to In Progress
IF pull_request.status = "draft" THEN project.status = "In Progress"
✓ Auto-rules configured
✅ Auto-Close Magic: When you merge a PR with "Closes #42" in the commit message, GitHub automatically closes issue #42 and updates the project board!

Branch Naming Conventions (Azure Boards)
# Format: <type>/<workitem-id>-<description>

# Examples:
feature/AB-1234-add-payment # Feature implementation
bugfix/AB-5678-fix-login-timeout # Bug fix
chore/AB-9012-update-dependencies # Chore/maintenance
refactor/AB-3456-service-layer # Refactoring
docs/AB-7890-api-documentation # Documentation

# Git workflow:
git checkout -b feature/AB-1234-add-payment
# Make changes...
git commit -m "feat: payment processing AB#1234"
git push origin feature/AB-1234-add-payment
# Create PR, which auto-links to AB#1234

Azure Boards automatically recognizes the work item ID (AB#1234) in your branch name and commit messages, creating automatic links.

Common Exam Traps
❌ TRAP: Epic and Feature are the same
Reality: Epic is strategic (6-12 months), Feature is tactical (1-3 months). Features roll up to Epics.
❌ TRAP: WIP limits are optional
Reality: WIP limits prevent team overload and force finishing work before starting new work (Kanban principle).
❌ TRAP: Story Points = Hours
Reality: Story points measure complexity/effort, not time. 8 points ≠ 8 hours. Use velocity to estimate sprints.
❌ TRAP: GitHub Projects and Azure Boards are the same
Reality: Azure Boards is enterprise-grade with sprints, velocity, test plans. GitHub Projects is simpler, integrated with code.
❌ TRAP: Branch naming doesn't matter
Reality: Branch names should include work item ID (AB-1234) for automatic linking and traceability.
Key Exam Questions to Prepare For
Q: When should you use WIP limits?
A: Always on Kanban boards to prevent overload and enforce completion before starting new work.
Q: What's the difference between Sprint capacity and velocity?
A: Capacity is available hours (team size × sprint days). Velocity is actual story points completed (historical average).
Q: How do you link GitHub issues to Azure Boards work items?
A: Use branch naming (AB-1234) and commit messages (AB#1234). Some teams use third-party integrations for two-way sync.
Q: What's the minimum Definition of Done?
A: Code reviewed, tests written, documentation updated, deployed to staging. No shortcutting!
Exam Pro Tip: Questions often focus on "what should the team do" scenarios. Remember: WIP limits prevent chaos, velocity predicts capacity, and traceability enables compliance.
