Debt Accumulation Impact on Velocity

Types of Technical Debt

Code Debt

Messy implementation, duplicated logic, poor abstractions, tight coupling

Impact: Hard to modify safely

Test Debt

Low coverage, no unit tests, brittle E2E tests, missing edge cases

Impact: Regressions hide in production

Documentation Debt

Outdated README, missing API docs, no architecture diagrams, no runbook

Impact: Onboarding takes weeks

Architecture Debt

Monolithic core, poor service boundaries, no clear data flow

Impact: Cannot scale independently

Dependency Debt

Outdated libraries, deprecated APIs, security vulnerabilities

Impact: Security risk, incompatibility

Infrastructure Debt

Manual deployments, no CI/CD, inconsistent environments

Impact: Unreliable deployments

⚠️ Balance Required: Some debt is necessary for speed, but unchecked debt kills productivity. The goal is conscious debt management, not zero debt.

New Code vs Overall Code

ℹ️ Clean as You Code: SonarQube tracks new code separately. Focus on keeping NEW code clean (100% coverage goal) while refactoring legacy code gradually.
Quality Gate: New Code
- Coverage > 80% ✓
- Duplications < 3% ✓
- Code Smells = 0 ✓
- Hotspots reviewed ✓

Quality Gate: Overall
- Coverage > 50% ✓ (legacy code drags this down)
- Bugs < 50 ✓

Code Coverage Deep Dive

Coverage measures what percentage of code is executed by tests. But not all coverage is equal!

Types of Coverage

Type What It Measures Example
Line Coverage Percentage of lines executed 80% = 8 of 10 lines hit by tests
Branch Coverage All if/else paths taken if (x > 0) → test both true AND false
Function Coverage All functions/methods called Called at least once in tests
Statement Coverage Each statement executed Very similar to line coverage

Branch Coverage Matters Most

Example: 100% line coverage but 0% branch coverage (DANGER!)

function processUser(user) {
  if (user.age >= 18) { // Line tested
    return "Adult";// Line tested
  }
  return "Minor";// Line NOT tested
}

Test: processUser({age: 20})
✓ Line coverage: 80% (4/5 lines)
✗ Branch coverage: 50% (only tested age >= 18, not < 18)

Coverage Realistic Targets

Unit Tests

70-85% coverage realistic

Fast, isolated, comprehensive

Goal: Branch coverage ≥80%

Integration Tests

40-60% coverage realistic

Slower, broader, fewer paths

Goal: Key flows covered

E2E Tests

20-40% coverage realistic

Slow, full stack, critical user journeys

Goal: Happy path + errors

Combined

Aim for 70%+ overall

Unit + Integration + E2E pyramid

Goal: Confidence in quality

Mutation Testing Concept

Coverage alone is misleading. Mutation testing asks: "Are your tests actually testing?"

Original: if (x > 5) { ... }
Mutation 1: if (x >= 5) { ... } ← Test catches? (Good!)
Mutation 2: if (x > 4) { ... } ← Test catches? (Good!)
Mutation 3: if (true) { ... } ← Test catches? (Bad! Test is weak)

✓ High mutation score = tests are thorough
✗ Low mutation score = tests are superficial

Azure DevOps Code Quality Integration

Embed code quality gates into your CI/CD pipeline to prevent low-quality code from reaching production.

SonarQube in azure-pipelines.yml

trigger:
  - main

pool:
  vmImage: 'ubuntu-latest'

steps:
  - task: SonarQubePrepare@5
    inputs:
      SonarQube: 'SQ-Production'
      scannerMode: 'MSBuild'
      projectKey: 'myproject'

  - task: SonarQubeAnalyze@5
    inputs:
      jdkversion: 'JAVA_HOME'

  - task: SonarQubePublish@5
    inputs:
      pollingTimeoutSec: '300'

  - task: PublishCodeCoverageResults@1
    inputs:
      codeCoverageTool: 'cobertura'
      summaryFileLocation: '**/coverage.xml'

Pull Request Decoration

SonarQube automatically comments on PRs with quality issues:

❌ SonarQube Quality Gate Failed

  • 5 new bugs introduced (file.ts:42, file.ts:127, ...)

  • Coverage dropped 8% (68% → 60%)

Review in SonarQube Dashboard

Quality Gate as Pipeline Gate

Block merge if quality gate fails:

- task: SonarQubePublish@5
  inputs:
    pollingTimeoutSec: '300'
    failWhenQualityGateFails: true ← Fails pipeline

If QG fails → Pipeline fails → Cannot merge

bash — test & publish

Measuring & Reducing Technical Debt

Quantify debt, prioritize reduction, and build a culture of quality.

SQALE Method: Measuring Debt in Hours

SQALE converts issues into estimated remediation time:

Issue TypeSeverityEstimated Effort
Code SmellMinor5-10 minutes
Duplicated CodeMinor15-30 minutes
High ComplexityMajor1-2 hours
Bug (Logic Error)Critical2-4 hours
VulnerabilityCritical2-3 hours
Architecture IssueBlocker8-40 hours

Example: 340 code smells × 10 min avg = 57 hours of debt

Technical Debt Ratio

Debt Ratio = Remediation Cost / Codebase Size

Example:
- 57 hours debt / 1,200 hours codebase = 4.8% debt ratio

0-5%: Healthy (manageable)
5-10%: Concerning (needs attention)
10%+: Critical (impacting velocity)

Strategies for Reducing Debt

Boy Scout Rule

"Leave the code cleaner than you found it"

On every commit, fix one minor smell. Over time, accumulated impact is massive.

Definition of Done

Enforce quality in acceptance criteria

- Tests written + passing
- Coverage maintained
- No new smells introduced

Tech Debt Sprint

Dedicate 1 sprint per quarter to debt

20-30% of capacity focused on high-impact, high-effort items

Debt Tracking Backlog

Backlog items for debt reduction

Prioritize by impact:effort ratio. Pull when sprint has capacity.

AZ-400 Exam Tips: Technical Debt & Quality 🎯

Technical Debt Quadrants

Key distinction: Reckless/Deliberate (bad), Prudent/Deliberate (acceptable), Inadvertent (learning)

Exam Q: "Team takes shortcut to hit deadline, plans refactor later?" = Prudent/Deliberate (OK)

Remember: Some debt is necessary for speed

SonarQube Quality Gates

Metrics tested: Bugs, Vulnerabilities, Code Smells, Coverage, Duplications, Complexity

Exam Q: "How to fail pipeline on low coverage?" Use Quality Gate with failWhenQualityGateFails: true

PR Decoration: SonarQube auto-comments on PRs with findings

Code Coverage Strategy

Branch coverage > Line coverage: Test all conditions, not just lines

Realistic targets: 70% unit, 50% integration, 30% E2E

Exam Q: "100% line coverage guarantees quality?" FALSE! Branch coverage matters

Azure Integration

Tasks: SonarQubePrepare, SonarQubeAnalyze, SonarQubePublish, PublishCodeCoverageResults

Order matters: Prepare → Build → Analyze → Publish

Exam Q: "Block merge on QG fail?" Use failWhenQualityGateFails in Publish task

Measuring Debt

SQALE: Converts issues → hours of remediation effort

Debt Ratio: Debt hours / Codebase size = % debt

Healthy: <5%, Concerning: 5-10%, Critical: >10%

Reducing Debt

Boy Scout Rule: Leave code cleaner = incremental improvement

Definition of Done: No new debt per feature

Priority matrix: High impact/low effort = do first

Common Exam Traps

❌ "High coverage = high quality" — Coverage is necessary but not sufficient!

❌ "Skip SonarQube in early projects" — Quality gates prevent debt accumulation

❌ "Technical debt is always bad" — Strategic debt for speed is acceptable

❌ "SonarQube works without QG setup" — Gate must be configured to fail pipeline
