🔷 Module 5: Git Hooks & Enforcement

What Are Git Hooks?

Git hooks are scripts that Git executes automatically at specific points in the development workflow. Think of them as airport security checkpoints — you can't proceed without passing through them.

Client-Side Hooks

Run on developer's local machine. Can be bypassed with --no-verify.

Runs before commit is finalized

Modifies the commit message

Validates commit message format

Runs after commit succeeds

Server-Side Hooks

Run on the Git server. Cannot be bypassed — enforced for all developers.

pre-receive
Validates incoming push
update
Per-branch validation
post-receive
Triggers after push accepted
ℹ️ Key Difference: Server hooks are mandatory for all developers. Client hooks can be skipped with git commit --no-verify.
✅ Airport Analogy: Just like you can't board a plane without passing security, you can't commit code without passing git hooks. Some checkpoints (TSA/pre-commit) are on your local machine and can theoretically be bypassed. Others (airport security desk/server hooks) are mandatory and enforced for everyone.

Writing Git Hooks

Pre-commit Hook (ESLint)

Run linting and code formatting checks before allowing a commit.

#!/bin/bash
# .git/hooks/pre-commit
# Prevent commits with linting errors

FILES=$(git diff --cached --name-only --diff-filter=ACM)

if [[ $FILES != "" ]]; then
  npx eslint $FILES
  if [[ $? != 0 ]]; then
    exit 1
  fi
fi

Install:

cp hooks/pre-commit .git/hooks/pre-commit
chmod +x .git/hooks/pre-commit
Commit-msg Hook (Conventional Commits)

Validate that commit messages follow the Conventional Commits format.

#!/bin/bash
# .git/hooks/commit-msg
MSG=$(cat $1)

# Check if message matches: type(scope): description
if ! [[ $MSG =~ ^(feat|fix|docs|style|refactor|test|chore)\: ]]; then
  echo "❌ Message must start with: feat|fix|docs|..."
  exit 1
fi
Pre-push Hook (Run Tests)

Prevent pushing code that fails unit tests.

#!/bin/bash
echo "Running unit tests before push..."
npm test

if [[ $? != 0 ]]; then
  echo "❌ Tests failed. Push cancelled."
  exit 1
fi

Team-Wide Hook Sharing with Husky

Problem: Hooks in .git/hooks/ aren't shared in Git. Solution: Use Husky to share hooks across the team.

// package.json
{
  "husky": {
    "hooks": {
      "pre-commit": "lint-staged",
      "commit-msg": "commitlint -E HUSKY_GIT_PARAMS",
      "pre-push": "npm test"
    }
  }
}
✅ Best Practice: Use Husky + lint-staged for client-side enforcement, and server-side hooks for mandatory enforcement.

Force Push Attack

Without server-side protection, a developer could force-push and rewrite history:

bash — ~/project
git push --force origin main
remote: error: [pre-receive hook] rejected push
remote: Force pushes are not allowed on main branch
fatal: the remote end hung up unexpectedly
🚫 Critical: Enable "Allow force pushes: No" on protected branches to prevent history rewriting attacks.
💡 Exam Tip: Know the difference between pre-receive (validates all refs), update (per-branch validation), and post-receive (triggers after acceptance, e.g., deployments).

Automatic Versioning with Semantic Release

Semantic Release Pipeline

Conventional Commits enable automatic versioning:

  • feat: increases minor version (1.2.0 → 1.3.0)
  • fix: increases patch version (1.2.0 → 1.2.1)
  • BREAKING CHANGE: increases major version (1.2.0 → 2.0.0)

semantic-release automatically:

  • Analyzes commits since last release
  • Determines new version number
  • Generates CHANGELOG.md
  • Creates Git tag
  • Publishes to npm/registry
✅ Exam Insight: Understand that Conventional Commits enable automation. A "feat" commit automatically triggers a minor version bump, a "fix" triggers a patch. This is how DevOps teams achieve continuous release.

Exam Tips for Git Hooks (AZ-400)

💡 Tip 1: Pre-commit hooks run on the developer's machine. They CAN be bypassed with git commit --no-verify. Server hooks CANNOT be bypassed.
💡 Tip 2: When exam asks "How do you ensure all developers follow commit standards?", the answer is SERVER-SIDE HOOKS or BRANCH PROTECTION RULES. Client-side hooks are for convenience.
💡 Tip 3: Distinguish these three server hooks:
pre-receive

Fires: Before any ref is updated

Sees: All refs being pushed

Use: Validate overall push

stdin: oldref newref refname

update

Fires: Once per branch

Sees: Single branch at a time

Use: Per-branch rules

args: refname oldrev newrev

post-receive

Fires: After push is accepted

Sees: All refs (read-only)

Use: Deploy, trigger CI/CD

Runs asynchronously

⚠️ Trap: Exam might ask "Which hook should prevent a push?" Answer: pre-receive or update. NOT post-receive (it runs after push already succeeded).
ℹ️ Exam Scenario: "You want to prevent commits with hardcoded secrets. Where do you put the check?" Answer: Both client (pre-commit hook with git-secrets) AND server (pre-receive hook). This is "defense in depth."
✅ Key Concept: Hooks + Branch Protection Rules + PR Reviews form the "three lines of defense" against bad code reaching main branch.
