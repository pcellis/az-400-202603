M01Create a Release Pipeline

CI vs CD

  • CI → artifact. CD → deploys artifact.
  • Same artifact through all environments
  • Config changes per env, code does not

Deployment Job

  • Use deployment: not job:
  • Tracks history in Environment
  • Supports deployment strategies
  • References environment by name

Artifact Sources

  • download: current — this pipeline
  • resources: pipelines: — another pipeline
  • Azure Artifacts feed — versioned packages
  • Container registry — images
🎯
Approvals on the Environment, not the stage. Classic = pre/post-deployment conditions on stage. YAML = checks on Environment. Exclusive lock prevents concurrent deployments to same environment.

M06Automate Inspection of Health

Health Signals

  • Availability — App Insights ping
  • Error rate — requests/failed
  • Response time — requests/duration
  • Custom metrics — business health

Rollback Speed

  • Feature flag — instant
  • Slot swap back — <1 min
  • K8s rollout undo — <2 min
  • Re-deploy artifact — 5-15 min

Integrations

  • Service hooks → Teams/Slack/PagerDuty
  • Work item #ID links to build
  • Deployment tracking on work items
  • DORA: Frequency/Lead/CFR/MTTR
🎯
Azure Monitor gate passes when NO active alerts exist. Fix alert noise before using Monitor as a gate. Service hooks = outbound from ADO to external tools (configured in Project Settings).

Multi-stage release with approval gate

stages: - stage: Build jobs: - job: BuildApp steps: - task: PublishPipelineArtifact@1 inputs: { artifact: drop } - stage: DeployStaging dependsOn: Build jobs: - deployment: Deploy environment: staging strategy: runOnce: deploy: steps: - download: current artifact: drop - task: AzureWebApp@1 inputs: { appName: myapp-staging } - stage: DeployProd dependsOn: DeployStaging condition: and(succeeded(), eq(variables['Build.SourceBranch'], 'refs/heads/main')) jobs: - deployment: Deploy environment: production # approval check set in ADO UI strategy: runOnce: deploy: steps: - task: AzureWebApp@1 inputs: { appName: myapp-prod }

Blue-Green — App Service slots

- task: AzureWebApp@1 inputs: appName: myapp-prod deployToSlotOrASE: true slotName: staging - task: AzureAppServiceManage@0 inputs: Action: Swap Slots WebAppName: myapp-prod SourceSlot: staging

RunOnce with smoke test + on:failure rollback

strategy: runOnce: deploy: steps: - task: AzureWebApp@1 inputs: { appName: myapp-prod } postRouteTraffic: steps: - script: curl -f https://myapp.azurewebsites.net/health || exit 1 on: failure: steps: - task: AzureAppServiceManage@0 inputs: Action: Swap Slots WebAppName: myapp-prod SourceSlot: production TargetSlot: staging

Canary — Kubernetes

strategy: canary: increments: [10, 25] deploy: steps: - task: KubernetesManifest@0 inputs: { action: deploy, manifests: manifests/ } postRouteTraffic: steps: - script: ./check-error-rate.sh on: failure: steps: - task: KubernetesManifest@0 inputs: { action: reject } success: steps: - task: KubernetesManifest@0 inputs: { action: promote }

Stage DAG — parallel + conditional

- stage: Build - stage: Test dependsOn: Build - stage: SecurityScan dependsOn: Build # parallel with Test - stage: DeployStaging dependsOn: [Test, SecurityScan] - stage: DeployProd dependsOn: DeployStaging condition: and(succeeded(), eq(variables['Build.SourceBranch'],'refs/heads/main')) - stage: Notify dependsOn: DeployProd condition: always() # run even on failure

🎯 Exam Flash Cards — LP3 All Modules

🎯
LP3 is one of the heaviest release strategy domains. 14 must-know questions.

🎯 Exam Watch — AZ-400

  • 🎯Q: Where do you configure human approval in YAML pipelines? → On the Environment resource in ADO UI — not in the YAML file.
  • 🎯Q: Blue-Green rollback — mechanism and speed? → Swap deployment slot back. Instant — old version is still warm.
  • 🎯Q: YAML canary strategy works with which environment type? → Kubernetes environments only. App Service canary = different mechanism.
  • 🎯Q: Feature flags vs canary — key difference? → Feature flags = user-level toggle, no re-deploy. Canary = infrastructure-level traffic split, reduces deployment risk.
  • 🎯Q: Feature flags ≠ A/B testing — why? → Canary/feature flags = risk reduction for new code. A/B = business metric comparison between experiences.
  • 🎯Q: Microsoft recommended IaC tool for Azure-only? → Bicep. Compiles to ARM JSON.
  • 🎯Q: SAST vs DAST — where does each run? → SAST in CI against source code. DAST in staging against live running app.
  • 🎯Q: Azure Monitor gate passes when — what condition? → NO active alerts matching the configured rule in the monitoring window.
  • 🎯Q: condition: always() on a stage — what does it do? → Stage runs regardless of upstream success, failure, or cancellation.
  • 🎯Q: Rolling strategy with 1 VM resource in Environment — behaviour? → Behaves like RunOnce. Rolling requires multiple resources to actually batch.
  • 🎯Q: Secret rotates in Key Vault — what pipeline change is needed? → None. KV-linked variable group fetches fresh values at next pipeline start automatically.
  • 🎯Q: Required template check — what does it prevent? → Any pipeline deploying to the protected environment without extending the approved template.
  • 🎯Q: Fastest rollback requiring zero re-deployment? → Feature flag disable (instant) or slot swap back (<1 min).
  • 🎯Q: Commit message #1234 in Azure DevOps — what happens? → Work item 1234 auto-links to the build. Work item shows deployment status across environments.
