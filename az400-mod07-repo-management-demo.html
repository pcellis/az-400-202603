Git Large File Storage (LFS)

Git is designed for source code (text), not large binary files. LFS solves the repo bloat problem.

The Problem: Repo Bloat

Without LFS (Traditional Git)

❌ Large binaries stored in git history

❌ Every clone downloads full history (2.3 GB)

❌ Slow clones, pushes, and pulls

❌ Difficulty pruning history

Example: PSD file (500 MB) × 10 commits = 5 GB history

With LFS (Recommended)

✓ Pointers stored in git (text files, ~200 bytes)

✓ Large files stored on LFS server

✓ Only needed files downloaded on checkout

✓ Repo size: 45 MB + separate LFS storage

Same 10 commits PSD = 45 MB git + LFS server

LFS Architecture Diagram

Developer → git push → Git Repo (pointers) + LFS Server (files)

When you git pull: Git downloads pointer, then LFS downloads actual file

Setup Walkthrough

bash — ~/project
$ git lfs install
Git LFS initialized.
✓ Hooks installed

$ git lfs track "*.psd" "*.zip" "*.jar"
Tracking *.psd
Tracking *.zip
Tracking *.jar
Updated .gitattributes
✓ Track patterns added

$ git add .gitattributes
$ git add design.psd large-file.zip
$ git commit -m "add design and artifacts"
[main abc1234] add design and artifacts
✓ Files tracked with LFS

$ git push origin main
Uploading LFS objects: 100%
✓ LFS server received files

Pointer File Format

.gitattributes
*.psd filter=lfs diff=lfs merge=lfs -text
*.zip filter=lfs diff=lfs merge=lfs -text
*.jar filter=lfs diff=lfs merge=lfs -text
design.psd (what's actually stored in git)
version https://git-lfs.github.com/spec/v1
oid sha256:4d967a2422f6
size 512000000

Purging Secrets from History

The absolute worst scenario: developer commits an API key, token, or credential to git. History is permanent—deletion doesn't help!

The Danger: Why Simple Deletion Fails

bash — ~/project
$ git log --all --full-history
abc1234 Remove API key from .env
def5678 Add production config
SECRET_KEY=sk_live_super_secret_12345
✗ Secret still exists in commit def5678 history!

$ git show def5678:.env
SECRET_KEY=sk_live_super_secret_12345
✗ Anyone with git history can extract the secret
🚫 CRITICAL: Secret is ALREADY compromised. Rotate/revoke it IMMEDIATELY on the service before touching git history!

Solution 1: git filter-repo (Recommended)

Rewrite entire git history to remove sensitive files.

bash — ~/project
$ pip install git-filter-repo
✓ Installed

$ git filter-repo --path .env --invert-paths
Filtering commits: 100%
Rewriting commits: 100%
✓ .env removed from all history

$ git push origin --force --all
⚠️ Force push to notify all collaborators
✓ Remote updated with cleaned history

Solution 2: BFG Repo Cleaner

Faster alternative for specific patterns.

bash — ~/project
$ java -jar bfg.jar --delete-files .env
Removed .env from 5 commits
History rewritten
✓ Done (usually faster than filter-repo)

$ git reflog expire --all --expire=now
$ git gc --prune=now --aggressive
Repacking repository...
✓ Garbage collected

After Purging: All Collaborators Must Re-clone

⚠️ Impact: Force push breaks all existing local clones. Team must run: git clone <repo> to get cleaned history. Rebasing existing PRs will fail!

Prevention: Secrets Never Enter Git

.gitignore

Never commit secrets:

.env
.env.local
*.key

Monorepo vs Multi-repo Architecture

Repository structure shapes team dynamics, deployment speed, and code sharing.

Side-by-Side Comparison

Monorepo (Single Repo)

One repo for entire organization

✓ Easier refactoring across services

✓ Shared code & libraries in same repo

✓ Atomic commits across services

✓ Single CI/CD setup

❌ Very large repo (scale issues)

❌ Harder permission boundaries

❌ All services use same deploy cadence

Users: Google, Microsoft, Meta

Multi-repo (Multiple Repos)

Separate repo per service/team

✓ Small, focused repos

✓ Clear permission boundaries

✓ Independent deploy cadence

✓ Easier to open-source

❌ Cross-repo refactoring painful

❌ Code duplication

❌ Dependency versioning challenges

Users: Most startups & enterprises

Decision Matrix: When to Use Each

FactorPrefer MonorepoPrefer Multi-repo
Team size50+ engineers<20 engineers
Code sharingHeavy (shared libs)Light (via packages)
Services couplingTightly coupledLoosely coupled
Deploy frequencyAll togetherIndependent
PermissionsCoarse (org-wide)Fine-grained (per team)
CI/CD complexityHigherLower

Azure DevOps Monorepo: Path-Based Triggers

Trigger pipelines only when specific service changes:

trigger:
  branches:
    include: [main]
  paths:
    include: [/services/api-gateway/*]
    exclude: [/services/api-gateway/docs/*]

# Only changes in /services/api-gateway/ trigger this pipeline
# Changes in /services/worker/ are ignored

Monorepo Tooling: Nx, Turborepo

Nx (JavaScript/TypeScript)

Task orchestration across monorepo

  • nx run-many --target=test (run tests on changed)
  • nx affected --target=build
  • Dependency graph visualization

Turborepo (JavaScript/TypeScript)

Fast monorepo build system

  • Caching across CI systems
  • Parallel task execution
  • Remote caching for teams

Hybrid Approach: Monorepo Per Domain

Best of both worlds:

organization/
  payments-mono/ (1 repo: billing-service, invoice-service)
  analytics-mono/ (1 repo: events-collector, dashboard)
  infra-mono/ (1 repo: terraform, k8s configs)

# Teams own their domain, can refactor freely within it
# Each monorepo has independent CI/CD & permissions

Version Strategies: Caret (^) vs Exact (=)

StrategyExampleAllows UpdatesUse Case
Caret (^)^1.2.31.2.3 to <2.0.0 (minor + patch)Most dependencies (default)
Tilde (~)~1.2.31.2.3 to <1.3.0 (patch only)Stable libraries
Exact (=)=1.2.3Only 1.2.3 (no updates)Critical/risky deps
Latest (*)*Any version (dangerous!)Never use in production

Dependency Lock Files

package.json

"express": "^4.18.0"
"lodash": "^4.17.0"

# Declares intent, not exact versions

package-lock.json

"express": "4.18.2"
"lodash": "4.17.21"

# Locked to exact versions
# npm ci uses lock file
ℹ️ Best Practice: Commit lock files to git! Ensures reproducible builds across CI/local/production.

AZ-400 Exam Tips: Repository Management 🎯

Permissions Mastery

Know the roles: GitHub (Read, Triage, Write, Maintain, Admin) and Azure DevOps (Reader, Contributor, Build Admin, Project Admin)

Exam Q: "What role allows pushing but not merging?" Answer: Write (GitHub) or Contributor (ADO)

Protected branches: Always require 2 reviewers + CODEOWNERS + status checks

LFS vs Regular Git

When to use LFS: Binary files (PSD, ZIP, JAR, ISO)

When NOT: Source code, JSON, YAML (git is optimized for these)

Exam Q: "Repo is 2.3 GB with one PSD file. How to reduce?" Answer: Use LFS

Remember: LFS requires git lfs install first!

Secret Purging Methods

Tools: filter-repo (modern), BFG (faster), git filter-branch (legacy)

Critical First Step: ROTATE the secret on the service immediately!

After purging: All team members must re-clone (force push breaks history)

Prevention: .gitignore, GitHub secret scanning, pre-commit hooks

Monorepo Tricks

Path-based triggers: Azure Pipelines can trigger on file changes in /services/api/ only

Decision factor: Team size + code sharing = monorepo; loosely coupled = multi-repo

Exam Q: "Team of 8 microservices, independent deploy?" Multi-repo is simpler

Tooling: Nx, Turborepo for monorepo optimization

Dependency Management

Vulnerability scanning: npm audit, pip audit, dotnet list package

Lock files MUST be committed: npm ci uses lock file for reproducibility

Dependabot: Auto-creates PRs for updates (configure schedule)

Exam Q: "Ensure same versions in CI and local?" Answer: Commit lock file

Common Traps

❌ "Deleting a file removes it from git history" — FALSE! Must use filter-repo

❌ "LFS works without git lfs install" — FALSE! Hook is essential

❌ "Admin can bypass branch protection" — PARTIALLY TRUE (force push)

❌ "CODEOWNERS is optional" — FALSE! Can be required by branch rules
